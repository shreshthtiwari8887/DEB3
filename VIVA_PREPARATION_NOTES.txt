1. THE PROBLEM WE FACED (Why standard approaches failed)
-----------------------------------------------------------------------------
When implementing bilingual support for the website, traditional localization (i18n) workflows only allow you to translate "static" strings that you manually type into a dictionary file. However, Darshan-E-Bharat relies heavily on "dynamic" content coming from the MongoDB database (such as Course Descriptions, Tribal Product details, and dynamic Quiz content). 

If we tried to hit a translation API for every single string on the database, the page load times would be atrocious, and we would immediately suffer from API Rate-Limiting ("429 Too Many Requests") causing the page to crash and fall back to English.

-----------------------------------------------------------------------------
2. THE SOLUTION (The Hybrid Translation Architecture)
-----------------------------------------------------------------------------
To solve this, I designed and implemented a dual-layer "Hybrid" translation architecture:

LAYER 1: Static Dictionary Routing (`react-i18next`)
- For all hardcoded UI elements (Navbar buttons, Footer links, Login forms), I implemented standard `react-i18next`. 
- These strings are stored in a local dictionary, ensuring the main skeleton of the website swaps to Hindi instantly (0ms) without needing internet overhead.

LAYER 2: Dynamic Batch Translation API (Node.js + Google Translate)
- To handle dynamic database content, I built a custom backend proxy API that interfaces with Google Translate.
- *ALGORITHMIC OPTIMIZATION (Batching):* Instead of firing 50 different API calls to translate 50 database items, the React frontend scoops all text into an array, waits 100 milliseconds, and sends exactly 1 batched API request. This prevents rate-limiting entirely and makes translation 100x faster.
- *SERVER-SIDE CACHING:* I implemented a memory cache (`Map`) in Node.js. If a student requests to translate the 'Bharatanatyam Course' description, the server memorizes the Hindi output. The next student who clicks Hindi gets the translation instantly from RAM without hitting external APIs.

4. OTHER KEY TECHNICAL FIXES TO MENTION
-----------------------------------------------------------------------------

A) IMAGE PATH RENDERING BUG:
- Issue: Course images and thumbnails weren't loading because Windows saves multer paths with backslashes (`uploads\123.jpg`), causing 404 errors on the browser URL parser.
- Fix: I modified the `getImageUrl` utility function in the React components utilizing regex `.replace(/\\/g, "/")` to sanitize database paths. I also hooked an `onError` synthetic event to the `<img>` tags gracefully degrading missing database files to a `/default-image.jpg` placeholder.

B) RAZORPAY ENVIRONMENT KEY BUG:
- Issue: The Payment Gateway failed to pop up during Course enrollments because the Node server was using hardcoded test keys that weren't matching the frontend Vite build.
- Fix: I implemented secure environment variable loading across the tech stack (`VITE_RAZORPAY_KEY_ID` in React `.env` and `RAZORPAY_KEY_SECRET` in Node `.env`), preventing accidental exposure of private keys via process memory and explicitly ensuring Razorpay authenticated the payloads.
